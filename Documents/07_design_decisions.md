# 07. 설계 결정 기록 (Design Decisions)

이 문서는 시스템을 설계하면서 내린 주요 결정들의 **배경, 검토한 대안, 최종 결정 이유**를 기록한다. 향후 설계를 변경하거나 새로운 결정을 내릴 때 참고용으로 사용한다.

각 결정은 ADR(Architecture Decision Record) 형식을 간략화해 정리.

---

## ADR-001. 두 층(Mode + Stage) 상태 관리

### 배경
같은 `LANE_DRIVING` 모드에서도 보행자 인식이 필요한 시점, 방해차량 인식이 필요한 시점, 어린이 보호구역 인식이 필요한 시점이 모두 다르다. 모드만으로 표현하면 인식기들이 자기가 켜질 시점을 알 수 없다.

### 검토한 대안
1. **모드 하나로 모든 걸 표현 (세분화된 모드)**: 모드 수가 폭증, control 로직 복잡화
2. **인식기가 자체적으로 활성화 시점 판단**: 인식기마다 미션 흐름을 알아야 함, 결합도 증가
3. **모드(control용) + 스테이지(인식·전환용) 두 층 분리** ← 선택

### 결정
Mode + Stage 두 층으로 분리. control은 모드만 보고 어떻게 움직일지 결정, 인식기는 스테이지에서 파생된 활성 인식기 목록을 보고 일할지 말지 결정.

### 효과
- control 단순화 (모드 7개만 처리)
- 인식기 활성화 규칙이 `STAGE_PERCEPTIONS` 한 곳에 모임
- 같은 모드를 여러 스테이지가 공유 가능

## ADR-002. 인식 결과를 통합 메시지로 발행

### 배경
인식기마다 토픽을 따로 만들면 토픽 개수가 9개 이상으로 늘어남. 또한 인식 담당자가 한 명이라 자기 관리하에 묶는 것이 합리적.

### 검토한 대안
1. **인식기마다 별도 토픽**: 토픽 9개+, main에서 콜백 9개 관리
2. **단일 배열 타입 토픽** (예: `float32[]`): 인덱스 외워야 하는 매직 넘버 문제
3. **단일 커스텀 메시지 토픽** ← 선택

### 결정
하나의 커스텀 메시지(`Perception`)에 모든 인식 결과를 필드로 묶어 `/ericar/perception` 단일 토픽으로 발행. 비활성 필드는 0/false로 채움.

### 효과
- 토픽 수 감소 → 관리 단순화
- 모든 인식 결과의 timestamp가 동일 → 동기화 문제 없음
- 필드명으로 의미 전달 → 가독성

### 주의사항
- 새 인식기 추가 시 메시지 변경 필요 → 메시지 패키지 리빌드 필요
- 비활성 인식기는 명시적으로 0/false로 채워야 함 (이전 값 남으면 안 됨)

## ADR-003. 정지(stop_request)를 모드 대신 플래그로

### 배경
정지가 발생하는 경우는 여러 가지: 보행자 감지, 빨간불, 좌회전 신호 대기 등. 각 경우마다 별도 모드를 만들면 모드가 폭증하고, 정지 해제 후 어느 모드로 복귀할지 복잡해진다.

### 검토한 대안
1. **각 정지 상황마다 별도 모드** (예: `STOP_PEDESTRIAN`, `STOP_RED_LIGHT`): 모드 수 증가
2. **정지를 모드와 직교하는 플래그로** ← 선택

### 결정
`/ericar/control_state.stop_request` 플래그로 정지 명령. control은 플래그가 true면 모드와 무관하게 `speed=0`을 출력.

### 효과
- 모드 수 안정 (7개 유지)
- 어떤 모드에서든 정지/재개 가능
- main이 "정지해야 할 이유"만 판단하면 됨

## ADR-004. 좌회전 offset도 같은 토픽에 발행

### 배경
좌회전 시점에 main이 control에 angle을 직접 발행하는 방식도 가능하나, 그러면 control이 모드별로 angle 계산 방식을 따로 가져야 함.

### 검토한 대안
1. **좌회전 모듈이 직접 angle 발행**: control 우회, 속도/PID 등을 좌회전 모듈도 알아야 함
2. **별도 토픽 `/ericar/left_turn_cmd`**: control이 토픽 두 개 보고 모드별로 선택해야 함
3. **차선/라바콘과 동일한 `driving_offset` 토픽에 발행** ← 선택

### 결정
좌회전 모듈이 IMU yaw 에러를 offset 형태로 환산해 `/ericar/driving_offset` (source=2)로 발행. control은 모드와 무관하게 동일 PID 처리.

### 효과
- control 단순화 (offset 처리 로직 단일화)
- 좌회전 모듈은 offset 계산만 책임 (속도는 모드로 control이 결정)
- B 담당자(차선/라바콘) 책임으로 통합

### 트레이드오프
- 좌회전이 offset이라는 추상화에 맞아야 함 → IMU yaw 기반이라 자연스러움

## ADR-005. main이 차선 변경 결정, B가 offset 제공

### 배경
방해차량 추월 시 차선 변경 결정을 어디서 할지가 문제. 후보는 인식 담당, B 담당, main.

### 검토한 대안
1. **인식 담당이 결정**: 인식과 결정이 섞임
2. **B가 결정**: B는 offset 발행에 집중하는 게 좋음
3. **main이 결정** ← 선택

### 결정
main이 `target_lane`을 발행해 B에게 어느 차선 기준 offset을 줄지 알림. 차선 변경 완료는 B가 판단해 main에 알림 (책임 분리).

### 효과
- 결정은 main, 실행은 B
- 두 책임이 깔끔하게 분리됨

## ADR-006. 모드를 string 대신 uint8 + 상수

### 배경
ControlState 메시지에서 모드를 어떻게 표현할지.

### 검토한 대안
1. **string** ("LANE_DRIVING" 등): 디버깅 편함, 오타 위험
2. **uint8 enum** (메시지 내 상수): 오타 방지, 디버깅 시 매핑 필요 ← 선택

### 결정
ControlState.mode는 uint8, 상수는 메시지 파일 내부에 `uint8 MODE_LANE_DRIVING = 2` 형식으로 정의. 디버깅용 string은 `/ericar/current_stage`(별도 토픽)로 발행.

### 효과
- 오타로 인한 버그 방지
- 메시지 크기 절약
- 디버깅은 별도 string 토픽으로

## ADR-007. ericar_msgs 별도 패키지 분리

### 배경
ROS2에서 커스텀 메시지는 별도 패키지로 빼는 것이 정석. 의존하는 패키지가 늘어나도 메시지 패키지 한 곳만 의존하면 됨.

### 결정
`ericar_msgs` 패키지를 분리해 4개 메시지(Perception, DrivingOffset, ControlState, ActivePerceptions)를 모음.

### 효과
- 모든 노드가 메시지 패키지 하나만 의존
- 메시지 변경 시 리빌드 범위가 명확

---

## ADR-008. 토픽 QoS: best-effort + depth=10

### 배경
30Hz 실시간 메시지에서 reliable QoS는 오버헤드. 실시간성과 최신성이 신뢰성보다 중요.

### 결정
모든 주기 발행 토픽은 best-effort + KEEP_LAST + depth=10.
변경 시 발행 토픽도 동일하나, 늦게 구독한 노드를 위해 변경 시 연속 발행 또는 1Hz heartbeat 추가 권장.

### 효과
- 낮은 통신 오버헤드
- 메시지 한두 개 누락은 다음 발행으로 회복

### 주의사항
- 한 번만 발행하면 안 되는 토픽(target_lane 등)은 추가 안전장치 필요

---

## ADR-009. 30Hz 발행 주기 통일

### 배경
각 노드가 다른 주기로 발행하면 동기화 문제 발생.

### 결정
주기 발행 토픽은 모두 30Hz로 통일.

### 효과
- 노드 간 데이터 흐름 일정
- 시뮬레이터 frame rate에도 적합 (보통 카메라 30fps)

### 예외
- 변경 시 발행 토픽 (target_lane, active_perceptions, current_stage)은 주기 발행 안 함
- /xycar_motor는 시뮬레이터 규약에 따름 (대개 동일)

---

## ADR-010. 차선 미검출 시 valid 플래그 없음

### 배경
차선이 일시적으로 안 보일 때 처리 방식.

### 검토한 대안
1. **valid 플래그**: control이 처리 분기 필요
2. **B가 내부적으로 처리**: control 단순화 ← 선택

### 결정
DrivingOffset에 valid 플래그 없음. B가 내부적으로 이전 값 활용, 점진적 감쇠 등으로 처리.

### 효과
- control 로직 단순
- 차선 미검출 처리 일관성 (B 담당자가 모든 케이스 책임)

---

## ADR-011. lap 카운트는 출발선(체스판) 기반, main이 관리

### 배경
3 lap 완주 판정 기준.

### 결정
- 출발선 인식(`start_line_detected`)을 인식 담당이 발행
- main이 카운트
- 출발선 위치는 트랙 다이어그램상 경로선택 신호등 바로 앞

### 효과
- lap 판정이 명확한 단일 지점 기반
- main이 single source of truth

---

## ADR-012. lap == 3 종료는 시뮬레이터에 위임

### 배경
3 lap 완주 후 명시적 종료 로직이 필요한지.

### 결정
시뮬레이터가 자동 종료하므로 별도 `FINISH` 스테이지 없음. 안전을 위해 lap == 3 도달 시 stop_request=true 권장.

### 효과
- 스테이지 수 절약
- 시뮬레이터 규약을 그대로 활용

---

## ADR-013. 신호등 enum 5단계 (0~4)

### 배경
신호등 상태를 어떻게 표현할지.

### 결정
```
0: 인식 안됨/대상 아님
1: 빨강
2: 주황 (또는 황색)
3: 초록 (직진)
4: 좌회전
```

### 효과
- 직진 신호와 좌회전 신호 구분
- "인식 안 됨"과 "초록 아님"을 0과 1~2로 구분

## ADR-014. 3-state 필드 (police_car_status, obstacle_faster_lane)

### 배경
"없음"과 "아직 판단 안 함"을 구분하지 않으면 main이 잘못된 판단을 할 수 있음.

### 결정
3-state로 표현:
- 0: 미판단/N/A
- 1: 첫 번째 값 (있음, 1차선)
- 2: 두 번째 값 (없음, 2차선)

### 효과
- main이 "판단 완료 후"에만 분기 가능
- false positive/negative 방지

## ADR-015. 패키지 3개 + 메시지 1개 = 총 4개

### 배경
3명이 각자 책임 영역을 나눠 작업.

### 결정
- `ericar_msgs`: 공용 메시지
- `ericar_perception`: A 담당
- `ericar_driving`: B 담당 (차선/라바콘/좌회전)
- `ericar_main_control`: C 담당 (main + control)

### 효과
- 패키지 = 담당자 명확
- 메시지는 모두가 의존하는 별도 패키지

## ADR-016. 좌회전 두 번째 트리거는 인식 신호(`shortcut_exit_trigger`)

### 배경
지름길에서 두 번째 좌회전을 언제 시작할지.

### 검토한 대안
1. **시간 기반**: 가장 단순, 정확도 낮음
2. **거리 기반**: IMU 필요, 정확도 중간
3. **차선 패턴 인식 (차선 끝남 등)** ← 선택

### 결정
인식 담당이 차선 패턴(차선 끝남)을 인식해 `shortcut_exit_trigger=true`를 발행. main은 이 신호로 좌회전 시작.

### 효과
- 위치 정확도 높음
- 구체적인 인식 방법은 구현자가 결정 가능

---

## ADR-017. 좌회전 완료 알림은 B → main

### 배경
좌회전이 끝났음을 누가 어떻게 알릴지.

### 결정 (잠정)
B가 IMU yaw 도달을 판단해 별도 토픽(`/ericar/left_turn_done` 등)으로 main에 알림.

### 효과
- main이 IMU를 직접 다룰 필요 없음
- B에 책임 집중

### 미정 사항
- 토픽 이름, 메시지 타입은 협의 후 결정

---

## ADR-018. 좌회전을 별도 Python 파일로 분리 (옵션)

### 배경
좌회전 로직이 일반 차선/라바콘 로직과 다름.

### 결정 (제안)
B 패키지 내부에 `left_turn_offset.py`를 따로 두고 모듈로 분리. driving_node가 import해서 사용.

### 효과
- 책임 분리 (좌회전 로직만 격리)
- 테스트하기 쉬움

### 자유도
실제 구현자가 결정 (별도 파일/노드/내부 함수 모두 가능)

---

## ADR-019. Header 포함

### 배경
모든 커스텀 메시지에 timestamp를 포함시킬지.

### 결정
모든 커스텀 메시지에 `std_msgs/Header header` 포함.

### 효과
- 디버깅 시 timestamp 추적 가능
- 향후 멀티 센서 동기화 시 활용 가능
- 발행자가 stamp만 채우면 되어 비용 적음
